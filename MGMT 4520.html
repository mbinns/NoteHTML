Systems and System Thinking

  • System and systems thinking are two different things
  • System:
    • A set of interrelated components that work together for a common cause
    • A description of a system
      • Components
      • Common Goal
      • Interrelationships how they work together
    • Example:
      • An Assembly line
        • An Assembly line is an example of a system because it has multiple components such as the people putting the parts on to the item being assembled. And each person is adding on the next part until the goal of a complete object is reached
    • Representing a system using a framework:
      • Input->Process->Output->Feedback->Process
      • Inputs:
        • Parts of the system that are responsible for bringing in data
      • Process:
        • Components that would manipulate or change the data that has been inputed
      • Output:
        • Components that are holding/present the finalized data
      • Feedback:
        • Components that are a kind of output component that inform whether the previous steps have function correctly
    • Example:
      • Withdrawing money from an ATM
      • Input:
        • Keypad:
          • Entering dollar amount
          • Entering Pin number
        • Card reader:
          • Insert Card
        • User:
        • Account Number:
          • If the user has multiple accounts
      • Process:
        • Checking account balance
        • Verifying card
      • Output:
        • Money
        • Receipt
        • Debit card returned
      • Feedback:
        • Receipt
        • Keypad beeping
        • Money you receive
  • Systems Thinking:
    • A process of understanding how the parts of a system impacts the whole
    • A problem solving approach that looks at the whole system to possibly identify which part(s) of the system where the problem could occur
    • A set of habits and practices based on the philosophy that components of any system can best be understood how they are related to one another and to the system

The Context of Key Stakeholders

  • Management Information Systems:
    • Supports day to day operations and decision makeup
    • Things you might want to track:
      • How many products are being made
      • Attendance/Who is working
      • Keeping track of damages
      • ETC.
    • Output:
      • Summary and exception reports
    • Goal:
      • Do things right in order to achieve a pre-decided outcome
  • TPS:
    • Collects, process, store, maintain and distribute all the data generated in business transactions
    • ERP (Enterprise Resource Planning Systems):
      • Interpreting TPS data
    • Output:
      • Organizational databases
  • DSS (Decision Support Systems):
    • Long term forward thinking strategic planning
    • Goal:
      • What is the right thing to do, I.E make a larger profit/Make more products/ETC.
    • Combine Internal and External data
  • EIS (Executive Information Systems):
    • Special type of DSS
    • Support Senior executives in decision making by providing tailored real-time data specific to each executive
  • Expert Systems:
    • Information Systems that mimic the advice giving abilities of experts:
      • http://www.WebMD.com is an example of this
  • Communication and Collaborative systems:
    • Facilitate interaction between people:
      • Google docs is an example
  • Office automation systems:
    • A fucking computer

The Context of Key Stakeholders

  • Stakeholders:
    • The people that are actually involved (Team)
    • Anyone who has a stake in the project
  • Information Workers:
    • White Collar Workers
      • Job is to create, collect, process, distribute or use information in some fundamental manner
    • Example:
      • Database Admin
      • Statistician/Analyst
      • CFO
      • Stockbroker
  • Knowledge Workers:
    • Subset of information workers
    • Apply specialized Subject matter expertise or knowledge to info to create new knowledge, innovations, solve problems and make decisions
    • Example:
      • Economist
      • Grad students
      • Entrepreneur
      • Analyst
      • CFO
      • Stockbroker
      • Database Admin
  • Business Stakeholders:
    • Owners:
      • High level Executives
      • Typically Sponsors or owners of the system being built
      • Duties:
        • Approve finances
        • Mobilizing resources
        • Championing the system:
          • Talking to people internally and externally to get them motivated and excited about the new system.
        • Face the music
    • Users:
      • People who will actually be using the system to get work done
      • Involving the users early will get them on-board with the system and make them WANT to use the system (Minimize user resistance)
      • Internal Users:
        • Working within the company
      • External Users:
        • Working from outside the company but are legally employed, or not
        • Example:
          • Customers
          • Contractors
          • Employees
          • Guests
          • Suppliers
        • Remote Users:
          • Working from a distance, but in the same location
        • Mobile Users:
          • Working from a distance, in different locations
      • Duties:
        • Testing and feedback
        • Requirements definition
        • Define the problem
  • Technical Stakeholders:
    • Designers
    • Builders
  • System Analyst Stakeholders:
    • Works with both the Business side and Technical Side and brings their opinions and views in-line with one another

Information System Architecture

  • High level framework for viewing an IS from different perspectives
    • Helps to unify the system for describing ANY information system that is being built
  • Three Key Components:
    • ALL information systems have these components
    • Data/Knowledge/Information:
      • What data will be brought into the system
      • What data will be produced as output
    • Process Component:
      • Tells you how the data will move through the system and how the data will be operated on
    • Communication/Interface Component:
      • How the system will relay the information outside of its self
      • To whom the system will provide data to

Building Blocks

  • IS Architecture:
    • Provides the three key components of an information system
      • Data
      • Process
      • Communication
  • You cross the key components with the 4 different perspectives of stake holders:
    • Stakeholders:
      • Business focused
        • Users
        • Owners
      • Technology focused
        • Designers
        • Builders
    • This will provide you with 12 building blocks of any information system (3 key components * 4 different stake holders)
  • Building blocks
    1. Owners perspective of data:
      1. Scope of data
      2. Data Vision
    2. Users perspective of data:
      1. More detailed view of data and how it is formatted/used
      2. Define requirements
      3. Think of it as line items on a memo/files in a filing cabinet
    3. Designers perspective of data:
      1. Concerned with the technologies involved
      2. Translates data requirements into designs
    4. Builders perspective of data:
      1. Actual technologies
      2. Converting designs
      3. Represent data in language like SQL
    5. Owners perspective of process:
      1. Scope
      2. Vision
      3. Not detail interested
      4. Define business events:
        1. Student requests course
        2. Student registered for course
    6. Users perspective of process:
      1. Specify Process Requirements
      2. Workflows
    7. Designers perspective of process:
      1. What technology must be implemented to achieve this
      2. Design layout/structure that depict workflows
      3. Software specs/What will be used
    8. Builders perspective of process:
      1. Implementing the technology
      2. ADE's (Application dev environment)
      3. Building functional prototypes
    9. Owners perspective of communication:
      1. What the system will interface with:
        1. users
        2. other systems
    10. Users perspective of communication:
      1. Interface requirements
    11. Designers perspective of communication:
      1. Two types of interaction
        1. system to system
        2. system to user
    12. Builders perspective of communication:
      1. Communicate with middle ware
      2. Integrate the interface with the app
  • Business drivers:
    • Continuous Process Improvement (CPI):
      • Continually making the product better
    • Total Quality Management (TQM)
      • Builds a culture where everyone is focused on improving cultures
  • Business function vs Business process:
    • Business function: has no defined start or end it operates for as long as the component needs it
    • Business process: Has a defined start and end point

System Life cycle

  • System Life Cycle:
    • This is the life of a system from conception to it becoming obsolete
    • Or in other words Everything a system will go through in its life time

System development Methodology

  • Specific steps or sequence of tasks needed to develop a system during the development phase of a systems life cycle
  • It is better to use a standard systems dev process methodology in an organization:
  • The employees will have more practice working within the system becoming efficient due to having lots of experience with it
  • The employees will be able to communicate between groups more effectively
  • Allows you to "debug" the system development process and possibly improve different aspects of it over the course of time "Better Quality Control"
  • Allows new members to get up to speed quickly
  • Helps org to easily move resources between system dev teams

Choosing inhouse or COTS SDM

  • Inhouse:
    • Pros:
      • More tailored methodology:
        • You can utilize the skill set of your employees better
      • If dev needs match the resources the company has then you can implement it cheaper
      • Cheaper to maintain because you have the internal people working on it
      • Also more flexible to accept updates
    • Cons:
      • More expensive to build and maintain if your company does not have the resources or skills to do so
      • May be misaligned to your business strategy/vision to build
      • Takes longer to develop using time/money/resources in "perfecting" the methodology
      • Improvement to the methodology will be slower
      • Outside contractors will have to learn your methodology
      • People who leave your organization don't have transferable skills

System Development Process v Project

  • How is managing a process different from managing a project:
    • Project management:
      • Is utilizing multiple processes to develop a system meeting the following criteria:
        • On Time
        • On Budget
        • First time Quality
    • Process management is about managing the internal sequence of events leading up to a goal:
      • Managing the steps involved in building the system

CCM 5 Levels of Maturity

  • What ever skills you gain at one level you retain at the next level
  • Gradation:
    • As the org moves from lvl 1 to level 5
      • Project management and process management capabilities are put into place and become more efficient
      • Reactive -> Proactive management of project and process
  • level 1:
    • "Anarchy"
    • Any given org will start at level one in their software development abilities
    • If the team has high skills and experience in software dev they may have good success
      • If there is low skill/exp then the team will fail
  • level 2:
    • "Repeatable"
    • First time Project management capabilities in place:
      • They are consistently monitoring three key things of project management:
        • Deadlines
        • Budget
        • First time quality
    • Since there is no consistent process in place everyone is doing their own thing
  • level 3:
    • Defined
    • First time Process management is implemented
      • Use a single systems development methodology across the whole org
      • Takes advantage of the pros of the single methodology
  • level 4:
    • "Managed"
    • Becoming proactive rather than reactionary in regards to project management
    • Capable of improving the practices they use for project management
  • level 5:
    • Becoming proactive rather than reactionary in regards to process management
    • Capable of improving the practices they use for process management

Core Principles in Process Development

  • There are ten core principles in process development
    1. Get system users involved:
      1. To eliminate miscommunication between users and the developers
      2. They will be more apt to take the system and use it
      3. Can help identify problems and successes with the system
    2. Employ a problem solving approach:
      1. This will help keep you on the rails
    3. Establish Phases and activities:
      1. This will break up the process into small subprocess and will define the steps you need to take to complete and move to the next step
    4. Document through the process:
      1. This is to give you the ability to retrace your steps
      2. Also give you the ability to improve the system you are working on
      3. You need to document because you will not be the only one looking at the system
    5. Establish Standards:
      1. Ensure that the org is using the same process for the systems they are developing
        1. This is because everything needs to be able to integrate when put together and if they all use different processes they will not work well with one another
    6. Manage the Process and Projects
      1. Manage:
        1. Time
        2. Budget
        3. Quality
      2. Also manage:
        1. How
    7. Justify system based on capital benefit:
      1. Cost/Benefit analysis
      2. This makes it easier for the owners to justify the cost of a system
    8. Don't be afraid to cancel or revise scope
      1. Defines the systems goals so that you know what it is and what it can and cant do
    9. Divide and conquer:
      1. You are more likely to build a system that users will like
      2. Minimize risk:
        1. This is because you only build little portions of the system and you can build on it/change it/cut losses
      3. Allows the users many more points of interaction
      4. What if you run out of money in the middle of the project:
        1. If you broke down the project into smaller subsystems at any given time you will have a usable system
    10. Design systems for growth and change

Problem Analysis Phase

  • This phase is about digging into the problems that were outlined in the scope definition phase
  • Questions you should ask your self
    • Are the problems really worth solving?
    • Is a new system really worth building?
    • This gives you the ability to have the creeping commitment where you only do bits and pieces at a time
    • Will the benefits of building the system outweigh the costs
  • Typically takes 6-8weeks

Key Purpose of P.A. Phase

  • 3 fold:
    • Understand the problem domain:
      • Refers to the context of the problem
      • This can be thought of as they current way the org does a task:
        • What processes they have in place
        • Flow of work
        • Flow of information
        • ETC.
      • Also the vocab that is going to be used in the current way of working
    • Cause and effect analysis:
      • This is of the preliminary problems/opportunities/directives from the Scope Def phase
      • Some of these problems will share a root cause and this phases will help you isolate the root cause of many of the problems you identified in the matrix
    • Develop system improvements objectives and constraints:
      • Improvement objectives
        • This is how this process would be improved
      • Constraints:
        • Each improvement is also accompanied by a constraint:
          • Constraints can be:
            • Time
            • Money
            • ETC.
        • This helps keep expectations grounded

Key Deliverables of the P.A. Phase

  • Deliverable 1
    • Understanding the problem domain:
    • Generally don't draw a lot of models in this phase, typically a tabular approach
