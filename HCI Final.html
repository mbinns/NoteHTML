Usability Testing

  • The extent in which a product can be used efficently and satisfaction in a specified context of use
  • Never test the user, only the product
  • Formative is durring development and summative is at the end of development
  • Usability Lab is the old school way to do it, it looks like an interrogation room
  • Difference between usability evaluaton and testing:
    • Evaluation is when experts look at the system
    • Testing is when users are brought into the system

Cognative walkthrough

  • An inspection method for evaluating the design of a user interface with special attention to how well the interface supprts exploratory learning. The eval us done by having a group of evaluators step through commonly used tasks. It can be preformed early in the design phase before preformance testing is possible
  • This is where experts are doing the test and you are not describing feelings
  • Trying to get into the midset of the novice users
  • You are trying to identify where users may get off task (Similar to thinkaloud)
  • Methods of a walkthrough:
    • Inspection method:
      • Have experts inspect a perticular interface or protype
      • Can be preformed in the early stages of design:
        • You get the most bang from your buck by doing this method
    • Ask specific questions as we stip through the interface:
      • Will users be trying to produce whatever effect a particular action has
      • Will users notice a corrct action is visible
      • Will the user associate the correct action with the effect to be achived
      • Will users understand feedback fter an action
  • These are really well suutied to first time users doing exploaratory learning with out any formal training. you are trying to imagine what a novice user might encounter when using a system
  • Requirements:
    • Who
    • What
    • Action sequence
    • Description of implementation of the interface
    • Usability experts, 3-4 is the ideal number
  • Assumptions:
    • A user may have some similar cutural background so that they would be familiar with a particular metaphore
    • Similar physical abilities
    • The user should have a goal defined
    • The user will explore
    • The users are going to choose somethong that seems reasonable
    • The user is able to interpret the response form the system correctly
  • Process:
    • Finding some collegues to participate in it with you
    • Define some tasks
    • Define the sequence of actions that ideally would allow users to accomplesh a particular goal
    • Take notes and keep track of things noted
    • Analyse data
  • Methodology:
    • As a walkthrough is being conducted recored you assumptions
    • Notes about side issues and design changes
    • Summary of results
    • Alter design to attempt to fix the problems noticed

Documentation

  • Can be useful for users, yourself, and people who may look at your code in the future
  • What is documentation:
    • Paper manuals, man pages, anything that provides information on the system
  • User support:
    • Help - problem specific
    • Documentation - general and system wide
  • Help and Doc:
    • Essential, but never a replacement for bad design
    • Most systems with reach features will require documentation
  • Manuals:
    • Cons:
      • Can be lost
      • Lots of irrelevant info
      • Takes up space
      • USERS DONT READ
    • Can be included on the device
  • Support requirements:
    • Availability:
      • Any time the users is at a system
    • Accurate and complete
    • Consistancy
    • Robustness:
      • Predictable and free of errors
    • Flexable
    • Unobtrusive
  • Tutorials:
    • Can provide a good conceptual model for how a system works
    • Convey the essential items that the users have to know in order to operate the system
    • essential interaction patterns that the user may have never seen before
  • Quick reference:
    • Like a cheet sheet
    • Good for experienced users
  • Context sensitive help:
    • Any information that you provide a user tha tis pertinent to the situatio that the user is in
    • Just in time contextual help
    • Be sure not to interrupt workflow
