Capability Maturity Model: CMM

  • Defines the levels of maturity for a software development org
  • Developed by software engineering institute
    • U.S defense dept.
  • Originally used to assess ability of government contractors to develop software
  • Levels rate the effectiveness of an orgs management
    • Level 1: Initial (Chaos)
      • Processes are chaotic, unrepeatable, undocumented
      • Orgs at this level can produce working software
        • Often exceeds budget and schedule
        • Successes often due to heroic efforts
      • Difficulty repeating past success
    • Level 2: Repeatable
      • Some aspects of processes are documented
      • Basic project management inplace
      • Organization can usually repeat success on projects similar in scope and domain to past efforts
    • Level 3: Defined
      • Software development processes are standardized and defined
        • Not just common in the head knowledge
      • Processes are improved over time
        • Will take time to assess what went well and what did not, then adjust the process to do better
      • Reviews are typically signed off on
    • Level 4: Managed
      • Management can control software development using quantitative techniques
        • Counting bugs, customer change revirews, etc.
      • Focus on gathering metrics and using them to make estimates for future projects
      • Processes are predictable
      • Orgs are much more likely to meet time / budget estimates
        • Using the quantitative estimates
    • Level 5: Optimizing
      • Focus on improving the software development process using metrics
        • Looking at the number of change requests generated for the projects they are working on and working to lower the numbers
  • Durring interviews:
    • Ask what type of process they follow
    • You can usually determine where the org stands by the response
    • Ask about project post-mortums to see if there is process improvement

Quality Assurance

  • Can refer to:
    • A set of activities designed to ensure software quality
    • A department in a software development orgs
  • Validation:
    • Are we building the right product
    • Focused on customer reqs
    • Collaborative activity: Customer is involved
    • Activity: Testing
  • Verification:
    • Does code meet spec
    • Internal activity: Dev/QA only

Verification

  • Three activities typically used:
    • Testing:
      • Execute program to check for bugs
      • Finds about 60-70% of the bugs in the code, it is impossible to find all of the bugs
    • Tracing:
      • Review source code to check for bugs
      • More eyeballs to check for more bugs
    • Formal Verification:
      • Prove source code correctly implements its spec
      • Most comprehensive checking and will find all of the bugs in the code, if the proof is correct
  • Combo of Testing and Tracing is the best way to find most bugs

The Role of Quality Assurance

  • An independent division in a development org
  • QA representatives paticipate in development process from start to finish
  • Effecting bug squashing involves collaborative processes between dev and QA

Testing

  • Addresses the ultimate question: "Does it work?"
  • Mindsets:
    • Test to show that the program meets reqs:
      • This is wrong because there is not a program that does not have bugs
    • Test to find bugs:
      • This is the correct mindset when going into testing
  • A successful test is a test that uncovers a bug

White Box v. Black Box

  • White box:
    • Internal perspective
    • Select input based on knowledge of implementation details
    • Debugger
    • Advantages:
      • Finds more bugs becuase you can design input that takes into account the design of the code
      • Finds the bugs that does not find the correct answer using the correct way, even if it find the right answer
      • More comprehensive
      • Can be done sooner
    • Disadvantage:
      • Must be done after the fact/can not be off loaded
      • More expensive
  • Black box:
    • External perspective
    • Does not use knowledge of implementation details
    • Feed in input observe output
    • Advantage:
      • Testing can be created in parrallel with the dev of the program if they have been included in the initial spec sheet convo
      • More through because they have a testing mindset
    • Disadvantage:
      • Cannot find the bugs were the output is correct but the method was wrong
  • Devs can do both black and white box
  • QA just does Black box

Testing Scopes

  • Unit testing:
    • isolates the component and feeds it input to make sure it performs correctly
  • Integration testing:
    • integrates the component into the system to make sure it performs correctly in a full environment
    • Testing parts of the whole system in a combined component environ
    • Can be automated
  • Validation testing:
    • Does it have the right features
    • Does it meet the specs
    • Running the whole system
    • More comprehensive that intergration testing
    • Never automated
  • Regression test:
    • Checks whether code modifications introduced bugs that were not present in the previous build
    • Done be QA

Testing Techniques

  • Agile QA Testing
  • Test driven dev:
    • Kkey agile dev technique
    • Write unit test before implementing functionality
    • Process (In Order):
      • Write spec for new method to be implemented
      • Create stub method (Empty)
      • Create unit test for method
      • Run test (obsureve failures)
      • Implement method
      • Run unit test
      • Degub if fails
    • Advantages:
      • Helps write the spec for the method percisely
      • Looks at the corner cases first
      • Gives you away to step through your code as soon as you have written it
      • Find more bugs
      • Psychologically motivating

Test Case Design

  • Consits of two things:
    • Input data
    • Expected results
  • Smallest number of cases that have the highest probablity of crashing the program
  • Approaches:
    • Specification based approach: (Black Box)
      • Design test inputs that conform to precondition
      • Compute expected results
    • Implementation based approach: (White Box)
